# Инструкция для агента: выкладка кода и деплой

Документ для передачи задачи другому ИИ или оператору. **Не вставляйте пароли, токены и приватные ключи в чат** — только в менеджер секретов, `.env` на сервере (не в git), SSH-ключи.

## 0. Контекст репозитория

- Корень: судейский SPA (Vite) → прод-артефакт каталог **`dist/`**.
- Подкаталог **`web/`** — Next.js 15 (standalone), API, `/sudyam`, админка.
- Полный чеклист окружения и компонентов: **[`docs/DEPLOY.md`](DEPLOY.md)**.
- Координация и CHANGELOG: **[`STATUS.md`](../STATUS.md)**.

## 1. Убедиться, что это Git-репозиторий

На машине с кодом (например `f:\3103` или клон с GitHub):

```bash
git status
```

- Если **«not a git repository»**: инициализировать или **клонировать** официальный репозиторий проекта, перенести изменения, затем продолжить.
- Задать **`origin`** (URL даёт владелец проекта, не придумывать):

```bash
git remote -v
# при необходимости:
# git remote add origin <HTTPS или SSH URL репозитория>
```

## 2. Локальная проверка перед пушем

Из **корня** репозитория:

```bash
npm ci
npm run test:gate
```

Минимум при нехватке времени: `npm run test:unit` + см. **[`docs/DEPLOY.md`](DEPLOY.md)** (сборки).

Сборки (как на сервере):

```bash
npm run build
cd web && npm ci && npm run build && cd ..
```

Оба шага должны завершаться **без ошибок**.

## 3. Коммит и push

```bash
git add -A
git status   # проверить состав коммита
git commit -m "Краткое описание изменений на русском или английском"
git push origin <имя-ветки>
```

Ветку (`main`, `master`, `develop`) согласовать с владельцем репозитория.

## 4. Деплой на сервер (общая схема)

Конкретика зависит от инфраструктуры. Агент **не должен** подключаться по SSH с паролем из чата; использовать **ключи**, **CI**, **панель хостинга**.

Типовые варианты:

1. **CI (GitHub Actions / GitLab / др.)**  
   После `git push` pipeline собирает артефакты и выкладывает на сервер.

2. **Сервер с git**  
   На сервере: `git pull` в каталог приложения, затем пересборка/рестарт процессов по **[`docs/DEPLOY.md`](DEPLOY.md)**.

3. **Статика SPA**  
   Содержимое **`dist/`** раздаётся nginx/Caddy/S3+CDN с корректными MIME (см. DEPLOY.md).

4. **Next.js**  
   `web/` → `npm run build`, запуск standalone или процесс из `.next/standalone`, **все prod env** из `web/.env.local.example` и внутренней документации — **только на сервере/в панели**, не в репозитории.

Обязательно на prod:

- применены миграции из **`migrations/`** по порядку;
- HTTPS и заголовки для iframe/CSP согласованы (см. DEPLOY.md).

## 5. Что передать агенту вместе с задачей

- URL **git-remote** и целевая **ветка**.
- Где хостится **статика** (`dist/`) и **Next** (`web/`) — один хост или два.
- Есть ли **CI** или только ручной pull на VPS.
- **Не передавать** root-пароли в чат; выдать доступ через ключ или задачу в CI.

## 6. Быстрый чеклист для другого ИИ

1. Открыть **[`docs/DEPLOY.md`](DEPLOY.md)** и **[`STATUS.md`](../STATUS.md)**.
2. Проверить `git status` / наличие `origin`.
3. Выполнить `npm run test:gate` (или оговорённый минимум).
4. Собрать корень и `web/` как в DEPLOY.md.
5. Закоммитить и `git push` на согласованную ветку.
6. Деплой выполнить **по принятой у команды схеме** (CI / SSH с ключом / панель), без секретов в логах чата.
